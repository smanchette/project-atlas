# Project Atlas v0.59.80 — UTF-8 Browser-Evidence Transport

v0.59.80 removes the only tracked browser-DOM transport that depended on host-shell text encoding. `capture_manual_browser_evidence.py` no longer reads standard input. A live capture must use an unpredictable raw-input file directly under the ignored `.runtime` directory, accompanied by the capture-side SHA-256 of its exact bytes. The helper independently reads and hashes those bytes before strict UTF-8 decoding. A mismatch fails as `browser_evidence_input_byte_mismatch`.

The input contract is BOM-free UTF-8. Invalid sequences, UTF-16, legacy code-page bytes, truncated multibyte characters, an explicit Unicode replacement character, stale input, an unapproved path or filename, a symlink, a missing file, or a malformed/mismatched hash fails closed. Raw bytes are never Unicode-normalized and line endings are never changed before the integrity comparison. Only after transport verification does the existing HTML identity layer resolve entities, normalize semantic text to NFC, and collapse semantic whitespace. Consequently a literal rendered en dash and `&#8211;` compare through HTML parsing, never through accidental transport mutation.

`wordpress_browser_evidence_transport.py` supplies the byte-only staging and independent consumption primitives. Staging uses an exclusive unpredictable file, writes and fsyncs the exact bytes, and limits permissions. The helper deletes live raw input in a `finally` block on success, validation failure, output failure, timeout interruption, or exception; cleanup failure is itself fail-closed. The PowerShell-to-Docker harness never reads DOM as text. It computes a binary file hash, invokes Docker with an argument array, exposes `.runtime` read/write only to the short-lived helper container so deletion can succeed, moves only normalized signed JSON, and cleans residual input/output in its own `finally` block.

The evidence schemas retain their meanings. Schema v1 still proves the corrected one-H1 Orlando page, and schema v2 remains restricted to the earlier duplicate-H1 state. Helper version `0.59.80` identifies the byte-exact capture implementation. The locked title, H1, canonical, image, metadata inventory, privacy, signature, and lifetime gates are unchanged; genuine Unicode or identity differences still fail.

All downstream consumers—deployment inspection, heading verification/reconciliation, installation, activation, plugin upgrade and cleanup, metadata lifecycle, rendering, cache, and recovery—accept the same signed evidence object and never launch the capture helper. The sole executable capture boundary and its operator documentation were corrected, so no consumer retains an alternate text-pipe path.

This source release does not capture production evidence, access WordPress, authorize or perform a plugin upgrade, enable rendering, purge cache, or mutate Atlas production data. Metadata Bridge remains 0.57.7 and its locked ZIP SHA-256 remains `ada4d97ea627a148d07fda809c1776a91a87d7a7e4957de3bece423a9bb80a62`.
